Matching fields:-
GDataV1 / Drive
title / originalFilename
exposure / exposureTime
focallength / focalLength
size / filesSize
model / cameraModel
timestamp / imageMediaMetadata.date
ABOVE conversion date = datetime.datetime.fromtimestamp(int(timestamp/1000 - 3600)).strftime('%Y-%m-%d %H:%M:%S')
     - 3600 is for timezone variation so not always applied TODO - look for timezone in metadata??

Note fileSize is relatively unique so is the best starting point for matches.

original file name and fileSize makes for a nearly unique match.
original file name and timestamp should make for nearly unique match.
But timestamp is harder to compare due variations in early cameras and timezone issues.

If we still get duplicates we would have to download the file with gdata and hash it.
Non exif files such as videos and gifs will not have date
More recent cameras have always unique filenames (but could clash between two cams with same naming scheme)
It seems that video files do not match their file size (maybe because of compression after setting size meta?)
  so these must be matched on date, filename.

Having implemented matching above I note the following:-
* Photos that started life in a drive folder outside of Google Photos and copied in will still appear to be in the original when scanning
so it is necessary to scan all of drive if you want to match all album references - not just Google Photos folder
* Some photos are simply not in Drive - this is a bug. If I open the photo in google photos and add a description I then see that photo
download into my Windows official Google Drive synced folder. So making a mod gets it rescanned and added into drive.
* Items generated by Google Photos are not on drive - these include MOVE.mv4 files, panoramas, collages, animation GIFs, EFFECTS,
